# 浏览器有哪些兼容问题，你封装过什么插件

这是一个非常实战的问题，考察的是工程化落地能力和解决历史遗留问题的经验。
作为前端开发者，我的回答分为两部分：首先是浏览器兼容性痛点与策略，其次是我封装过的通用插件/工具库案例。
#  第一部分：浏览器兼容性问题与应对策略
虽然现在现代浏览器（Chrome, Edge, Safari, Firefox）对标准的支持已经非常统一，但在实际业务中（尤其是面向国内复杂环境、政企项目或海外老旧设备），我们依然面临以下挑战：
### 1. CSS 兼容性问题
- Flexbox/Grid 的旧实现：iOS 9-10 和部分 Android 4.x 对 Flexbox 支持不完整（如 flex-shrink 计算错误，Grid 布局完全不支持）。
 - 对策：使用 PostCSS + Autoprefixer 自动添加前缀；对于 Grid，提供 float 或 inline-block 的降级方案（通过 @supports 检测）。
- 视口单位 (vw/vh) 与 iOS 安全区：iOS Safari 的地址栏伸缩导致 100vh 高度计算不准；刘海屏遮挡内容。
 - 对策：使用 JS 动态计算 --vh 变量；强制使用 env(safe-area-inset-bottom) 处理底部安全区。
- 滚动行为：position: sticky 在旧版 iOS 需要 -webkit- 前缀；overflow: scroll 在部分安卓机上滚动卡顿或失效。
 - 对策：使用 better-scroll 等成熟库替代原生滚动，或进行特性检测降级。
### 2. JavaScript 兼容性问题
- ES6+ 语法支持：IE11 不支持 let/const、箭头函数、Class、Promise、Async/Await 等。
 - 对策：Babel 是核心。配置 @babel/preset-env 根据 browserslist 自动转译。
- 新 API 缺失：Array.prototype.includes, Object.assign, fetch, IntersectionObserver 等在旧浏览器不存在。
 - 对策：引入 Core-js 和 Regenerator-runtime 进行 Polyfill。注意按需加载，避免包体积过大。
- 事件模型差异：IE8/9 的 attachEvent vs 标准的 addEventListener；事件对象 e 的获取方式不同。
 - 对策：现代开发通常直接放弃 IE8/9，若必须支持，则通过封装统一的事件工具函数抹平差异。
### 3. 渲染与行为差异
- iOS 日期解析：new Date('2023-01-01') 在 iOS < 13 中返回 Invalid Date（只支持 / 分隔）。
 - 对策：全局重写 Date 构造函数或使用 dayjs 等库统一处理格式化。
- 输入框焦点与软键盘：Android 和 iOS 弹出键盘时，页面布局被顶起或留白，fixed 元素错位。
 - 对策：监听 focus/blur 事件，手动调整 scrollTop 或使用 visualViewport API。
- 图片格式：WebP/AVIF 在旧 Safari/IE 不支持。
 - 对策：使用 <picture> 标签提供 fallback，或通过 CDN 根据 User-Agent 动态转换格式。
### 4. 我的兼容性策略总结
我不再盲目兼容所有浏览器，而是基于数据驱动：
1. 定义目标：根据业务用户画像（如：覆盖 95% 用户），制定 browserslist 配置。
2. 分层构建：利用 Modern Mode (ES Modules) 和 Legacy Mode (SystemJS/Babel) 下发不同代码包，让新浏览器跑新代码，旧浏览器跑转译代码。
3. 优雅降级：核心功能必须可用，非核心特效（如模糊背景、复杂动画）在低配设备上直接隐藏。

# 第二部分：我封装过的插件/工具库
在过往项目中，为了解决重复劳动和统一规范，我封装过多个高复用性的 npm 包或内部 SDK。以下是三个典型案例：
### 案例一：@my-org/request-sdk (基于 Axios 的企业级请求库)
- 背景：团队中每个项目都重复写 Axios 拦截器，处理 Token 刷新、错误码映射、Loading 状态逻辑不一致。
- 核心功能：
 1. 无感 Token 刷新：当接口返回 401 时，自动挂起后续请求，发起刷新 Token 请求，成功后重试队列中的请求，失败则统一跳转登录。
 2. 统一错误处理：将后端复杂的错误码（如 10001: 库存不足）转换为前端友好的 Toast 提示或抛出自定义 Error 对象。
 3. 防重复提交：内置 Request Debounce，相同 URL 和方法的请求在短时间内只发送一次。
 4. 类型安全：基于 TypeScript 泛型，传入 Response 类型即可自动推断返回值类型。
- 技术难点：处理并发刷新 Token 时的“惊群效应”（多个请求同时触发刷新），使用了单例模式 + Promise 队列来解决。
### 案例二：@my-org/mobile-hybrid-bridge (H5 与 Native 通信桥接)
- 背景：混合开发（Hybrid App）中，H5 需要调用摄像头、定位、支付等原生能力，不同 App 版本、不同平台（iOS/Android）的 - JSBridge 注入时机和协议不一致。
- 核心功能：
 1. 协议标准化：统一了 invoke(method, params, callback) 接口，屏蔽底层 WebViewJavascriptBridge 或自定义 schema 的差异。
 2. 环境检测与降级：自动判断是否在 App 内，若非 App 环境，自动 Mock 数据或提示“请在 App 内打开”。
 3. 回调管理：解决了原生回调丢失的问题，通过生成唯一 callbackId 映射本地 Promise，确保异步流程可控。
 4. 预加载机制：在 Bridge 对象注入前，将调用请求存入队列，注入完成后自动执行。
- 价值：将 H5 与原生对接时间从平均 2 天缩短到 2 小时，且消除了 90% 的通信 Bug。
### 案例三：@my-org/data-visualization-lite (轻量级大屏图表组件)
- 背景：项目需要展示大量实时数据大屏，引入 ECharts 全量包体积太大（~800KB+），且只需基础柱状图、折线图和简单的动画。
- 核心功能：
 1. 按需渲染：基于 Canvas 自研，仅实现业务需要的 3 种图表，体积控制在 30KB 以内。 
 2. 高性能动画：使用 requestAnimationFrame 实现流畅的数据更新动画，支持万级数据点的不重绘优化（局部脏矩形渲染）。
 3. 响应式适配：内置 ResizeObserver，自动根据容器大小重绘，适配各种大屏分辨率。
 4. 技术难点：在低端安卓机上保证 60fps，采用了对象池技术复用图形对象，减少 GC 压力。


# 总结
- 对于兼容性问题，我的态度是：“战略上重视（配置完善），战术上灵活（按需降级）”，不为了 1% 的用户牺牲 99% 的开发体验，但必须保证核心链路畅通。
- 对于插件封装，我的原则是：“源于业务，高于业务”。不仅仅是代码的抽取，更是对最佳实践、错误处理、类型安全和性能优化的固化，从而提升整个团队的研发效率和代码质量。
