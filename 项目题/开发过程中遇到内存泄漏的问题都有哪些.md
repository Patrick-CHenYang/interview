# 开发过程中遇到内存泄漏的问题都有哪些
在 JavaScript 这种带有自动垃圾回收（GC）机制的语言中，内存泄漏通常不是“忘记释放”，而是“无意中保留了不再需要的对象的引用”，导致 GC 无法回收它们。

在开发过程中，我遇到的内存泄漏主要集中在以下 5 大类场景：

### 1. 意外的全局变量 (Accidental Global Variables)
这是最基础但也最容易忽视的泄漏。
成因：在非严格模式（non-strict mode）下，如果在函数内给一个未声明的变量赋值，它会自动变成 window 对象上的属性。由于全局对象的生命周期等同于页面生命周期，这些变量永远不会被回收。
典型代码：
```js
function leak() {
  // 忘记写 var/let/const
  userData = { largeData: new Array(10000).fill('x') }; 
  // 等价于 window.userData = ...
}
```
解决方案：
   强制开启严格模式：'use strict';（现代构建工具如 Webpack/Vite 默认开启）。
   Lint 检查：使用 ESLint 的 no-implicit-globals 规则。

### 2. 定时器与回调未清理 (Timers and Callbacks)
setInterval 是内存泄漏的“重灾区”。
成因：定时器回调函数如果引用了外部作用域的变量（闭包），只要定时器还在运行，这些变量就无法被回收。即使组件已经卸载，如果定时器没停，它依然在后台运行并占用内存。
典型代码：
```js
// React 类组件或旧式写法
componentDidMount() {
  this.interval = setInterval(() => {
    // 引用了 this.state 或大对象
    this.updateData(); 
  }, 1000);
}
// ❌ 忘记在 componentWillUnmount 中 clearInterval
```
- 解决方案：
  - 在组件卸载（useEffect 的 cleanup 函数或 componentWillUnmount）时务必清除定时器。
  - 尽量使用 setTimeout 递归调用代替 setInterval，以便更好地控制执行流。

### 3. 闭包滥用与 DOM 引用 (Closures & Detached DOM Trees)
闭包本身不是泄漏，但错误的闭包会导致泄漏。
成因 A：大对象闭包：闭包持有了一个大对象的引用，而该闭包被长期存活的事件监听器引用。
成因 B：分离的 DOM 树 (Detached DOM)：JS 中持有一个 DOM 节点的引用，即使该节点已经从 DOM 树上移除（removeChild），只要 JS 变量还指着它，整个子树都无法回收。
场景：缓存了一个包含大量子节点的父节点引用，后来只操作了子节点，却忘了清空父节点引用。
解决方案：
仅在必要时使用闭包。
当 DOM 节点移除时，显式地将指向它的 JS 变量设为 null。
使用 Chrome DevTools 的 Memory -> Heap Snapshot 对比快照，查找 "Detached" 关键字。
### 4. 事件监听器未移除 (Event Listeners)
这在单页应用（SPA）中极其常见。
成因：在组件挂载时添加了全局事件监听（如 window.addEventListener('resize', ...) 或 document.addEventListener），但在组件卸载时没有移除。随着路由切换，组件反复创建销毁，监听器越积越多，且每个监听器都持有组件实例的引用。
典型代码：
```js
// Vue/React 组件
mounted() {
  window.addEventListener('scroll', this.handleScroll);
}
// ❌ 忘记在 unmounted/unmount 中 removeEventListener
```
- 解决方案：
 - 成对出现：addEventListener 和 removeEventListener 必须成对出现。
 - 使用 once 选项：如果只需要触发一次，使用 { once: true }。
 - 框架优势：利用 React useEffect 的 cleanup 或 Vue 的 onUnmounted 钩子自动管理。

### 5. 第三方库与缓存策略不当
成因：
无限增长的缓存：自己实现的缓存对象（如 Map 或普通对象）只增不减，没有 LRU（最近最少使用）淘汰机制。
第三方库实例未销毁：某些重型库（如 ECharts, Mapbox, D3）创建的实例内部维护了大量 DOM 和事件，如果只移除 DOM 而不调用库提供的 .dispose() 或 .destroy() 方法，会导致严重泄漏。
解决方案：
实现带容量限制的缓存策略。
严格遵守第三方库的生命周期管理文档，该销毁时必须销毁。
使用 WeakMap 和 WeakSet：它们持有的是“弱引用”，允许 GC 在只有弱引用存在时回收对象，非常适合做缓存键。

# 如何排查与定位？(面试加分项)
作为高级开发，不仅要会修，还要会查。我的排查流程如下：
1. 观察现象：Chrome Task Manager (Shift + Esc) 观察内存曲线是否呈“阶梯状”持续上升，且不下降。
2. Heap Snapshot (堆快照)：
 - 在 Chrome DevTools -> Memory 面板录制快照。
 - 执行操作（如切换路由、点击按钮）。
 - 再次录制快照。
 - 对比分析：选择 "Comparison" 视图，按 "Delta" 排序。重点看 Shallow Size 和 Retained Size 增长明显的对象。
 - 查看 "Retainers" 面板，找到是谁（哪个全局变量、哪个定时器、哪个监听器）持有了这些对象不放手。
3. Allocation Timeline (分配时间线)：实时查看内存分配情况，定位具体是哪一行代码在频繁分配内存且未被回收。

# 总结
内存泄漏的本质是“不该有的引用”。
避免它的核心心法是：谁创建，谁销毁；谁监听，谁移除；谁缓存，谁清理。
在现代框架（React/Vue）中，善用生命周期钩子和 useEffect 的清理机制，可以规避掉 80% 的常见问题。剩下的 20% 则需要我们对闭包原理和 GC 机制有深刻的理解。
